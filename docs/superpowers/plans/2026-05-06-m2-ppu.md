### Task 1: Per-dot clock skeleton (`PpuState`)

**Background.** The 2C02 PPU runs 262 scanlines × 341 dots per frame. Scanlines 0–239 are visible; 240 is post-render (idle); 241–260 is vblank (NMI fires on dot 1 of scanline 241); 261 is pre-render. Frames alternate parity: on odd frames, when rendering is enabled, the pre-render scanline is shortened by one dot (the "skipped tick at the end of pre-render"), producing 89341 dots instead of 89342. M2 puts the parity flip in this task; the actual skip lands in Task 2.

The `step()` API is called by `Bus::tick` exactly 3 times per CPU cycle (this is already wired in M1's bus). Each `step()` call advances one PPU dot.

```rust
//! Per-dot PPU clock: dot 0..340, scanline 0..261, frame parity.
//!
//! At M2 only the counters live here. Rendering side effects are added by
//! background.rs and sprite.rs.

use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub struct PpuState {
    /// Current dot within the scanline: 0..=340.
    pub dot: u16,
    /// Current scanline: 0..=261. 0..=239 visible, 240 post-render,
    /// 241..=260 vblank, 261 pre-render.
    pub scanline: u16,
    /// Even (`false`) / odd (`true`) frame parity. Toggles at the end of each frame.
    pub frame_parity: bool,
    /// Total frames completed since power-on (for tests / debugger).
    pub frames: u64,
}

impl PpuState {
    pub fn new() -> Self {
        Self { dot: 0, scanline: 0, frame_parity: false, frames: 0 }
    }

    /// Advance one dot. Wraps to next scanline at dot 340 and to next
    /// frame at scanline 261. Frame parity flips on every wrap to scanline 0.
    pub fn advance_dot(&mut self) {
        self.dot += 1;
        if self.dot > 340 {
            self.dot = 0;
            self.scanline += 1;
            if self.scanline > 261 {
                self.scanline = 0;
                self.frame_parity = !self.frame_parity;
                self.frames += 1;
            }
        }
    }
}

impl Default for PpuState {
    fn default() -> Self {
        Self::new()
    }
}
```

### Task 2: Frame parity skip on odd frames (pre-render dot 339→340)

**Background.** On odd frames, when rendering is enabled (PPUMASK bit 3 or 4 set), the pre-render scanline (261) skips its last dot — i.e., when `scanline == 261, dot == 339, frame_parity == true, rendering_enabled`, the next `advance_dot()` should jump straight to `(0, 0)` instead of going to `(261, 340) → (0, 0)`. ### Task 3: Loopy v/t/x/w internal registers — types + reset state

**Background.** The PPU has four internal registers, conventionally named after Loopy's seminal scrolling document:

- **`v`** (15 bits): the current VRAM address — used both for fetching during rendering and for the CPU-facing $2007 (PPUDATA) accesses.
- **`t`** (15 bits): the temporary VRAM address — staged through PPUSCROLL/PPUADDR writes, copied into `v` at well-defined moments.
- **`x`** (3 bits): the fine-X scroll offset — selects which of the 8 pixels in the shifters to emit each dot.
- **`w`** (1 bit): the write toggle — flips between the two-write halves of PPUSCROLL ($2005) and PPUADDR ($2006). Reset by reading PPUSTATUS ($2002).

The 15 bits of `v`/`t` decompose as:
```
yyy NN YYYYY XXXXX
||| || ||||| +++++- coarse X scroll (5 bits) — tile X
||| || +++++------- coarse Y scroll (5 bits) — tile Y
||| ++------------- nametable select (2 bits)
+++---------------- fine Y scroll (3 bits) — pixel Y within tile
```

This task introduces the types and reset values; later tasks (5, 7, 8) wire writes through them.

References:
- nesdev wiki — *PPU scrolling* (Loopy's doc) — https://www.nesdev.org/wiki/PPU_scrolling

```rust
//! PPU register file: external CPU-facing registers ($2000-$2007) and
//! Loopy internal registers (v, t, x, w).
//!
//! Decomposition of v/t (15 bits each):
//!   yyy NN YYYYY XXXXX
//!   ||| || ||||| +++++- coarse X scroll (5 bits)
//!   ||| || +++++------- coarse Y scroll (5 bits)
//!   ||| ++------------- nametable select (2 bits)
//!   +++---------------- fine Y scroll (3 bits)
//!
//! See nesdev wiki: https://www.nesdev.org/wiki/PPU_scrolling

use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Default, PartialEq, Eq, Serialize, Deserialize)]
pub struct Registers {
    /// PPUCTRL ($2000) — write only, latched.
    pub ctrl: u8,
    /// PPUMASK ($2001) — write only, latched.
    pub mask: u8,
    /// PPUSTATUS ($2002) — read clears bit 7 and the w toggle. Bits 5-7 only.
    pub status: u8,
    /// OAMADDR ($2003) — write only.
    pub oamaddr: u8,
    /// Internal: current VRAM address (15 bits).
    pub v: u16,
    /// Internal: temporary VRAM address (15 bits, staged via $2005/$2006).
    pub t: u16,
    /// Internal: fine X scroll (3 bits).
    pub x: u8,
    /// Internal: write toggle (1 bit).
    pub w: bool,
    /// Internal: PPUDATA read buffer for $0000-$3EFF reads.
    pub read_buffer: u8,
    /// Internal: open-bus latch for register reads (for unimplemented bits).
    pub open_bus: u8,
}

impl Registers {
    pub fn new() -> Self {
        Self::default()
    }

    /// Reset state per Mesen power-on snapshot. PPUCTRL/MASK/STATUS/OAMADDR
    /// all 0; v/t/x/w/buffers all 0.
    pub fn reset(&mut self) {
        *self = Self::default();
    }
}

    use super::*;
```

### Task 4: PPUCTRL ($2000) write — flags + nametable bits → t

**Background.** PPUCTRL is write-only, latched. Bit 7 enables NMI on vblank; bits 0–1 are the base nametable select (which goes into `t` bits 10–11); bit 2 selects PPUDATA increment (0 = +1 across, 1 = +32 down a column); bit 3 selects sprite pattern table base ($0000 or $1000); bit 4 selects background pattern table base; bit 5 selects sprite size (0 = 8×8, 1 = 8×16); bit 6 is the unused EXT pin (we ignore it). ### Task 7: PPUSCROLL ($2005) — two-write sequence, fine-X + coarse-X/Y

**Background.** PPUSCROLL is write-only, two-write sequence controlled by `w`. The first write (`w == false`) sets fine-X (3 low bits → `x`) and coarse-X (5 high bits → `t` bits 0–4). The second write (`w == true`) sets fine-Y (3 low bits → `t` bits 12–14) and coarse-Y (5 high bits → `t` bits 5–9). Each write toggles `w`. Reading PPUSTATUS clears `w` (already in Task 6).

Mnemonic for which bits go where: PPUSCROLL writes the two halves of the screen-pixel coordinate into `t`; the first byte carries X (with fine-X being the *low* 3 bits because pixels are scrolled finely on x), the second byte carries Y. PPUADDR (Task 8) writes the same bits but treats them as a VRAM address — confusingly, the layout is the same.

### Task 8: PPUADDR ($2006) — two-write sequence, t → v on second write

```rust
    /// First write: high 6 bits → t[13:8], t[14] cleared. Second write:
    /// low 8 bits → t[7:0]; then v ← t.

            self.v = self.t;
```

### Task 9: PPUDATA ($2007) — buffered read for $0000-$3EFF, immediate for palette, v auto-increment

**Background.** PPUDATA is read-write, accesses VRAM at address `v`, auto-increments `v` by 1 or 32 (per PPUCTRL bit 2) after each access. The read has a documented quirk: for addresses $0000–$3EFF, the read returns the *previous* contents of an internal buffer, then refills the buffer with the current address's data (a one-access read latency). For palette reads ($3F00–$3FFF), the read returns immediately — but the buffer is also refilled with what's at `v - $1000` (the underlying nametable mirror, since palette space mirrors $2F00–$2FFF in the buffer). Writes are always immediate.

During rendering (visible scanlines + pre-render with rendering enabled), $2007 access has documented "increment both x and y" behavior — accidental increments to coarse-X and fine-Y/coarse-Y. We implement the conservative version: outside of rendering it just adds the increment; during rendering we still just add the increment (the corruption case is rare and the test ROMs at M2 don't exercise it). A note for M11 / future: the "during rendering" corruption is a real hardware behavior that some demoscene ROMs depend on.

This task wires the PPUDATA register-side machinery (buffered read, address increment, palette-immediate path). The actual VRAM/palette/CHR access is supplied as a closure parameter, so the test can pass a synthetic bus reader without needing the full `Ppu` wired up. Task 13 closes the loop by routing the closure to real VRAM/palette/CHR.

```rust
    /// PPUDATA ($2007) read with a closure-supplied VRAM bus. Used both
    /// by `Ppu::cpu_read` (which provides a real bus closure) and by
    /// unit tests. The address is `v & 0x3FFF` (PPU bus is 14 bits).
    pub fn read_ppudata_with<F: FnMut(u16) -> u8>(&mut self, mut bus_read: F) -> u8 {
        let addr = self.v & 0x3FFF;
        let val = if addr < 0x3F00 {
            // Buffered: return previous buffer, refill with current.
            let prev = self.read_buffer;
            self.read_buffer = bus_read(addr);
            prev
        } else {
            // Palette: immediate. Buffer refilled with the nametable mirror.
            self.read_buffer = bus_read(addr - 0x1000);
            bus_read(addr)
        };
        self.v = (self.v.wrapping_add(self.vram_increment())) & 0x7FFF;
        val
    }

    /// PPUDATA ($2007) write with a closure-supplied VRAM bus.
    pub fn write_ppudata_with<F: FnMut(u16, u8)>(&mut self, val: u8, mut bus_write: F) {
        let addr = self.v & 0x3FFF;
        bus_write(addr, val);
        self.v = (self.v.wrapping_add(self.vram_increment())) & 0x7FFF;
    }
}
```

### Task 10: Nametable RAM (2KB) + mirroring helper

**Background.** The PPU owns 2KB of nametable RAM (the "VRAM" in classic NES parlance). The cartridge supplies a mirroring mode (Vertical / Horizontal / SingleScreen / FourScreen — see [`cartridge.rs:8-16`](crates/nies-core/src/cartridge.rs#L8-L16)) that determines how the 4KB virtual nametable space ($2000–$2FFF) maps onto the 2KB physical array.

Mirroring math: extract the 12-bit offset within $2000-$2FFF, then:
- **Vertical:** offset & 0x7FF — nametables 0/1 use the first KB, 2/3 mirror them.
- **Horizontal:** ((offset & 0x800) >> 1) | (offset & 0x3FF) — nametables 0/1 share the first KB, 2/3 share the second.
- **SingleScreen:** offset & 0x3FF — all four nametables mirror to the first KB.
- **FourScreen:** the cartridge supplies extra RAM; M2 panics on this case (no commercial NROM uses four-screen). Acknowledged limitation; M11+ revisits.

References:
- nesdev wiki — *Mirroring* — https://www.nesdev.org/wiki/Mirroring

```rust
/// Map a PPU address in $2000-$3EFF into a 0..2048 byte index in the
/// 2KB nametable RAM, applying the cartridge's mirroring mode.
///
/// Panics on `FourScreen` — that mode requires cartridge-supplied extra
/// RAM and is post-v1 (see M2 design spec §3.3).
pub fn nametable_index(addr: u16, mirroring: Mirroring) -> usize {
    let offset = (addr & 0x0FFF) as usize; // 0..4095
    match mirroring {
        Mirroring::Vertical => offset & 0x7FF,
        Mirroring::Horizontal => ((offset & 0x800) >> 1) | (offset & 0x3FF),
        Mirroring::SingleScreen => offset & 0x3FF,
        Mirroring::FourScreen => {
            panic!("FourScreen mirroring not supported at M2 (post-v1)")
        }
    }
}
```

### Task 11: Primary OAM (256B) — read/write helpers + power-on init

**Background.** OAM (Object Attribute Memory) holds 64 sprite entries × 4 bytes = 256 bytes total. Each entry is `[Y, tile_index, attributes, X]`:
- Byte 0: Y position minus 1 (so $00 means "top of screen, partly off-top").
- Byte 1: tile index (8×8) or tile pair (8×16 mode).
- Byte 2: attributes — palette (bits 0–1), priority (bit 5), flip H (bit 6), flip V (bit 7); bits 2–4 unused.
- Byte 3: X position.

OAM is initialized to a known pattern at power-on (Mesen uses 0xFF for all 256 bytes; some games rely on this). Reads via OAMDATA ($2004) follow OAMADDR; writes increment OAMADDR. During rendering, OAMDATA reads return $FF for dots 1-64 (when secondary OAM is being cleared); we wire that condition in Task 48.

### Task 12: Palette RAM (32B) + $3F1x mirror semantics

**Background.** The PPU has 32 bytes of palette RAM at $3F00-$3F1F. Layout: $3F00 is the universal background color; $3F01-$3F03, $3F05-$3F07, $3F09-$3F0B, $3F0D-$3F0F are the four background palettes (3 colors each); $3F11-$3F13, $3F15-$3F17, $3F19-$3F1B, $3F1D-$3F1F are the four sprite palettes (3 colors each). The "transparency" indices ($3F04, $3F08, $3F0C, $3F10, $3F14, $3F18, $3F1C) are physically only 4 bytes — $3F10/$3F14/$3F18/$3F1C *mirror* to $3F00/$3F04/$3F08/$3F0C. Write to $3F10 and you can read it back at $3F00.

Greyscale mask (PPUMASK bit 0) forces the low 4 bits of the read value to 0 (so palette indices snap to $00, $10, $20, $30).

The full $3F00–$3FFF range mirrors $3F00–$3F1F (so the 32 bytes appear 8 times). The mod-32 + the $3F1x→$3F0x mirror is one helper.

- nesdev wiki — *PPU palettes* §"Memory map" — https://www.nesdev.org/wiki/PPU_palettes

```rust
    if idx & 0b1_0011 == 0b1_0000 {

    idx
```

### Task 13: PPU public surface — `cpu_read` / `cpu_write` for $2000-$3FFF

```rust
impl Ppu {
    /// CPU-side register read at $2000-$3FFF. The address is mirrored
    /// down to the 8 register bytes via `& 0x7`.
    pub fn cpu_read(&mut self, mapper: &mut MapperKind, addr: u16) -> u8 {
        match addr & 0x7 {
            0 | 1 | 3 | 5 | 6 => self.regs.open_bus, // write-only
            2 => self.regs.read_ppustatus(),
            4 => self.oam.read(self.regs.oamaddr),
            7 => {
                let mirroring = mapper.mirroring();
                self.regs.read_ppudata_with(|a| {
                    if a < 0x2000 {
                        mapper.ppu_read(a)
                    } else if a < 0x3F00 {
                        self.vram.read(a, mirroring)
                    } else {
                        self.palette.read(a)
                    }
                })
            }
            _ => unreachable!(),
        }
    }

    /// CPU-side register write at $2000-$3FFF.
    pub fn cpu_write(&mut self, mapper: &mut MapperKind, addr: u16, val: u8) {
        self.regs.open_bus = val;
        match addr & 0x7 {
            0 => self.regs.write_ppuctrl(val),
            1 => self.regs.write_ppumask(val),
            2 => {} // PPUSTATUS is read-only
            3 => self.regs.oamaddr = val,
            4 => {
                self.oam.write(self.regs.oamaddr, val);
                self.regs.oamaddr = self.regs.oamaddr.wrapping_add(1);
            }
            5 => self.regs.write_ppuscroll(val),
            6 => self.regs.write_ppuaddr(val),
            7 => {
                let mirroring = mapper.mirroring();
                self.regs.write_ppudata_with(val, |a, v| {
                    if a < 0x2000 {
                        mapper.ppu_write(a, v);
                    } else if a < 0x3F00 {
                        self.vram.write(a, v, mirroring);
                    } else {
                        self.palette.write(a, v);
                    }
                });
            }
            _ => unreachable!(),
        }
    }

    /// Side-effect-free PPU register read for the debugger.
    pub fn cpu_peek(&self, _mapper: &MapperKind, addr: u16) -> u8 {
        match addr & 0x7 {
            0 | 1 | 3 | 5 | 6 => self.regs.open_bus,
            2 => (self.regs.status & 0xE0) | (self.regs.open_bus & 0x1F),
            4 => self.oam.read(self.regs.oamaddr),
            7 => self.regs.read_buffer, // approximation
            _ => unreachable!(),
        }
    }
}
```

### Task 18: NMI line — `take_nmi()` with rising-edge semantics

**Background.** Per design spec §3.1, NMI uses take-on-read edge semantics. The PPU asserts an internal `nmi_line` based on `(PPUCTRL bit 7) AND (PPUSTATUS bit 7)`; on a rising edge it sets `nmi_pending_take`. `take_nmi()` returns `true` and clears `nmi_pending_take` on call. The bus polls `take_nmi()` after each `step()`.

The level-vs-edge nuance: the line is `level = ctrl_bit_7 && status_bit_7`. A rising edge happens when `level` transitions from `false` to `true`. This can happen in two situations:
1. Vblank flag becomes 1 while NMI is enabled (the common case).
2. NMI bit becomes 1 while vblank flag is already set (the "late NMI" case, tested by `04-nmi_control.nes`). This is wired in Task 19.

This task handles case 1 only.

### Task 20: Bus wiring — propagate `take_nmi()` to `Cpu::nmi_pending`

**Background.** Currently `Cpu::nmi_pending` is set only by tests; nothing else writes it. Now we wire it: every time `Bus::tick` calls `ppu.step(mapper)`, it calls `ppu.take_nmi()` afterward and OR's the result into a bus-local pending flag. The CPU samples this flag at instruction boundaries.

The simplest design: `Cpu::nmi_pending` is set directly. But the CPU's step loop is "take instruction, then tick the bus 2-7 times depending on instruction." If we set `cpu.nmi_pending = true` *during* a tick, the CPU sees it at the next instruction boundary — which is correct.

The lifetime issue: `Bus::tick` doesn't have a reference to the CPU (the CPU drives the bus, not vice versa). Solution: the bus stores a `pending_nmi: bool` on itself; `Cpu::step` reads-and-clears it before checking `nmi_pending`. Or: `Bus::tick` stores the latch and `Cpu` reads it via `bus.take_pending_nmi()`.

We'll use the second pattern (pull, not push) to keep the bus -> cpu direction one-way.

### Task 21: PPUSTATUS read suppression at scanline 241 dot 0/1 boundary

**Background.** When the CPU reads PPUSTATUS at exactly the right moment around the start of vblank, the documented hardware behavior suppresses the NMI for that frame. Specifically:

- A read at scanline 241, dot 0 (the dot *just before* vblank would normally be set): suppresses the vblank flag for this frame entirely. The flag stays 0 even at dot 1.
- A read at scanline 241, dot 1: returns vblank=0 (the read won the race), and clears the flag (no NMI).
- A read at scanline 241, dot 2: returns vblank=1, clears the flag (NMI may have already fired or be about to).

This is the famous "race window" tested by `06-suppression.nes`, `07-nmi_on_timing.nes`, `08-nmi_off_timing.nes`.

Implementation: in `cpu_read` for $2002, check `self.state.scanline == 241 && self.state.dot == 0`. If so, set a "suppress this frame's vblank" flag that prevents the (241,1) flag-set in `step()`. At dot 1, the read returns the flag value just before clear — but the flag is set "as" the dot becomes 1, so the read at dot 1 must happen *after* the increment. Our step ordering gives dot 1 ↑ flag ↑ NMI edge — a read happens between two `step()` calls, so it sees the post-step state. Special case: if the read happens during the same dot 1 as the flag-set, we suppress. Easier model: dot 0 read sets a `vblank_suppress = true` flag for this frame; the flag-set at dot 1 of scanline 241 is gated by `!vblank_suppress`. Reset `vblank_suppress` at scanline 261 dot 1 (end of vblank).

References:
- nesdev wiki — *NMI* §"Race condition" — https://www.nesdev.org/wiki/NMI

### Task 25: 8-cycle fetch state machine — phase dispatch

**Background.** The visible-scanline render pipeline issues one PPU bus access per dot, in a repeating 8-dot pattern. For dots 1-256 (and again 321-336 for the next scanline's first two tiles):

| Dot mod 8 | Phase | Address | Latches |
|---|---|---|---|
| 1 | NT byte fetch | `0x2000 \| (v & 0x0FFF)` | `nt_byte` |
| 2 | (idle / completes NT fetch) | — | — |
| 3 | AT byte fetch | `0x23C0 \| (v & 0x0C00) \| ((v >> 4) & 0x38) \| ((v >> 2) & 0x07)` | `at_byte` |
| 4 | (completes AT fetch) | — | — |
| 5 | Pattern lo fetch | `bg_pattern_base + (nt_byte << 4) + ((v >> 12) & 7)` | `pat_lo` |
| 6 | (completes pat lo) | — | — |
| 7 | Pattern hi fetch | same + 8 | `pat_hi` |
| 8 | (completes pat hi) — coarse-X increment | — | shifters loaded |

The "completes" rows are because each PPU memory access takes 2 dots (1 dot to put the address on the bus, 1 dot to latch the data). For our purposes — bus-tick model where a `mapper.ppu_read(addr)` returns instantly — we can collapse each pair: do the work on the odd dot of each pair (1, 3, 5, 7).

This task introduces the dispatch only. The actual fetches land in Task 26.

References:
- nesdev wiki — *PPU rendering* §"Cycles 1-256" — https://www.nesdev.org/wiki/PPU_rendering

```rust
//! Background fetch pipeline: 8-dot repeating fetches feeding 16-bit
//! pattern shifters and 8-bit attribute latches.

use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Default, PartialEq, Eq, Serialize, Deserialize)]
pub struct Background {
    /// Latched nametable byte (current tile's pattern index).
    pub nt_byte: u8,
    /// Latched attribute byte (palette select for the 4×4-tile region).
    pub at_byte: u8,
    /// Latched pattern low byte (current tile, current row).
    pub pat_lo_latch: u8,
    /// Latched pattern high byte.
    pub pat_hi_latch: u8,
    /// 16-bit pattern shifters (low + high). High 8 bits hold the current
    /// tile's pixels; low 8 bits hold the next tile's pixels.
    pub shift_pat_lo: u16,
    pub shift_pat_hi: u16,
    /// 8-bit attribute shifters; each pixel sees its 2-bit palette select.
    pub shift_at_lo: u8,
    pub shift_at_hi: u8,
    /// Latched attribute bits for the *next* tile (loaded at dot 8 boundary).
    pub at_latch_lo: bool,
    pub at_latch_hi: bool,
}

impl Background {
    pub fn new() -> Self { Self::default() }

    /// Shift the pattern + attribute shifters one bit. Called every visible
    /// dot (1-256, 321-336 — i.e., when rendering is active and we're
    /// emitting pixels or fetching).
    pub fn shift(&mut self) {
        self.shift_pat_lo <<= 1;
        self.shift_pat_hi <<= 1;
        self.shift_at_lo = (self.shift_at_lo << 1) | (self.at_latch_lo as u8);
        self.shift_at_hi = (self.shift_at_hi << 1) | (self.at_latch_hi as u8);
    }

    /// Load the next tile's latched values into the low half of the
    /// pattern shifters; load the attribute latches from at_byte. Called
    /// at the dot-8 boundary of each fetch group.
    pub fn reload_shifters(&mut self) {
        self.shift_pat_lo = (self.shift_pat_lo & 0xFF00) | self.pat_lo_latch as u16;
        self.shift_pat_hi = (self.shift_pat_hi & 0xFF00) | self.pat_hi_latch as u16;
        // Attribute bits: which 2-bit palette select applies to the next tile?
        // Decoded from at_byte based on coarse Y bit 1 and coarse X bit 1
        // (which select among the 4 quads of the 32×32-px attribute block).
        // The selection is done by the caller (it has access to v); here we
        // accept the two bits directly via at_latch_{lo,hi}.
    }
}
```

### Task 26: Background fetches — NT, AT, pattern lo, pattern hi

**Background.** This task implements the four fetch types per the table in Task 25. Address formulas:

- **NT byte:** `0x2000 | (v & 0x0FFF)`. Routes through `Ppu`'s nametable accessor: $2000-$2FFF goes through VRAM with cartridge mirroring.
- **AT byte:** `0x23C0 | (v & 0x0C00) | ((v >> 4) & 0x38) | ((v >> 2) & 0x07)`. Decodes into the same nametable's attribute table.
- **Pattern lo:** `bg_pattern_base + (nt_byte as u16) << 4 + ((v >> 12) & 7)`. Goes through `mapper.ppu_read` (CHR space).
- **Pattern hi:** same + 8.

The attribute byte represents 2-bit palette selects for a 4×4-tile (32×32-pixel) region; the actual 2 bits for the *current* tile are picked based on `(coarse_y & 2)` and `(coarse_x & 2)` — bits 1 of coarse Y and X select among the four quads of the AT block.

References:
- nesdev wiki — *PPU rendering* §"Cycles 1-256" formulas — https://www.nesdev.org/wiki/PPU_rendering

**Files:**
- Modify: `crates/nies-core/src/ppu/mod.rs`

- [ ] **Step 1: Add fetch helpers to `Ppu`.**

```rust
impl Ppu {
    /// Compute the nametable byte address from v.
    fn nt_addr(&self) -> u16 {
        0x2000 | (self.regs.v & 0x0FFF)
    }

    /// Compute the attribute byte address from v.
    fn at_addr(&self) -> u16 {
        0x23C0
            | (self.regs.v & 0x0C00)
            | ((self.regs.v >> 4) & 0x38)
            | ((self.regs.v >> 2) & 0x07)
    }

    /// Compute the pattern table address for a given byte plane (lo or hi).
    fn pat_addr(&self, hi: bool) -> u16 {
        let base = self.regs.bg_pattern_table_base();
        let tile = self.bg.nt_byte as u16;
        let fine_y = (self.regs.v >> 12) & 7;
        base + (tile << 4) + fine_y + if hi { 8 } else { 0 }
    }

    /// Read a PPU bus address (CHR for $0000-$1FFF, VRAM for $2000-$3EFF).
    fn ppu_bus_read(&mut self, mapper: &mut MapperKind, addr: u16) -> u8 {
        let mirroring = mapper.mirroring();
        if addr < 0x2000 {
            mapper.ppu_read(addr)
        } else if addr < 0x3F00 {
            self.vram.read(addr, mirroring)
        } else {
            self.palette.read(addr)
        }
    }

    /// Pick the 2 attribute bits for the current tile from at_byte.
    fn at_bits_for_tile(&self) -> (bool, bool) {
        let coarse_x_hi = (self.regs.v >> 1) & 1; // bit 1 of coarse X
        let coarse_y_hi = (self.regs.v >> 6) & 1; // bit 1 of coarse Y
        let shift = ((coarse_y_hi << 1) | coarse_x_hi) * 2;
        let bits = (self.bg.at_byte >> shift) & 0b11;
        (bits & 1 != 0, bits & 2 != 0)
    }
}
```

- [ ] **Step 2: Add the dispatched fetch in `step`.**

Replace `Ppu::step` body. The fetch happens during dots 1-256 and 321-336 of visible + pre-render scanlines (0-239 and 261). Pattern: at dot mod 8 == 1, 3, 5, 7, do the corresponding fetch. At dot mod 8 == 0 (and dot ≥ 9), reload shifters and increment coarse X.

```rust
pub fn step(&mut self, mapper: &mut MapperKind) {
    let rendering = self.regs.rendering_enabled();
    let scanline = self.state.scanline;
    let dot = self.state.dot;
    let visible_or_pre = scanline < 240 || scanline == 261;

    // Background pipeline runs during dots 1-256 and 321-336 of visible/pre-render.
    if rendering && visible_or_pre && (1..=256).contains(&dot) || (321..=336).contains(&dot) {
        // Shift first (so pixel emit at dot N reads pre-shift state set up by dot N-1).
        if (1..=256).contains(&dot) || (321..=336).contains(&dot) {
            self.bg.shift();
        }
        match dot % 8 {
            1 => self.bg.nt_byte = self.ppu_bus_read(mapper, self.nt_addr()),
            3 => self.bg.at_byte = self.ppu_bus_read(mapper, self.at_addr()),
            5 => self.bg.pat_lo_latch = self.ppu_bus_read(mapper, self.pat_addr(false)),
            7 => self.bg.pat_hi_latch = self.ppu_bus_read(mapper, self.pat_addr(true)),
            0 => {
                // dot 8, 16, ..., 256, and 328, 336 — reload shifters + increment coarse X
                let (al, ah) = self.at_bits_for_tile();
                self.bg.at_latch_lo = al;
                self.bg.at_latch_hi = ah;
                self.bg.reload_shifters();
                // coarse X increment is in Task 30
            }
            _ => {}
        }
    }

    // Pixel emit (Task 29) happens here too — left as a stub for now.

    self.state.advance_dot_with_rendering(rendering);

    // Vblank flag transitions
    if self.state.dot == 1 {
        match self.state.scanline {
            241 => {
                if !self.vblank_suppress { self.regs.status |= 0x80; }
            }
            261 => {
                self.regs.status &= 0x1F;
                self.vblank_suppress = false;
            }
            _ => {}
        }
    }
    self.update_nmi_line();
}
```

### Task 30: Per-dot pixel emit during dots 1-256 of visible scanlines

**Background.** Now that the shifters are running and the v register is correct, each visible dot emits one background pixel into the framebuffer. Pixel selection: pick bit `15 - x` (where x is fine X) of `shift_pat_hi:shift_pat_lo` and combine with the same bit of `shift_at_hi:shift_at_lo`. Result is a 4-bit palette index: `[at_hi, at_lo, pat_hi, pat_lo]`.

The 4 bits index into the 16-byte background palette region ($3F00-$3F0F). Index 0 (any palette with pat bits 00) is "transparent" → use the universal background color at $3F00.

For visible scanlines (0-239) at dots 1-256, write the 6-bit palette color (read from palette RAM) into framebuffer at `[scanline][dot - 1]`.

**Leftmost 8-pixel clipping** (PPUMASK bit 1 == 0 for background): for dots 1-8, output the universal background color regardless of pattern data.

References:
- nesdev wiki — *PPU rendering* §"Pixel output" — https://www.nesdev.org/wiki/PPU_rendering

**Files:**
- Modify: `crates/nies-core/src/ppu/mod.rs`

```rust
impl Ppu {
    /// Emit one background pixel into the framebuffer at (scanline, dot-1).
    fn emit_bg_pixel(&mut self) {
        let scanline = self.state.scanline as usize;
        let dot = self.state.dot as usize;
        if scanline >= 240 || !(1..=256).contains(&dot) {
            return;
        }

        // Show background bit + leftmost-8 clip
        let show_bg = self.regs.show_bg();
        let clip_left = !self.regs.show_bg_left8() && dot <= 8;

        let palette_addr = if !show_bg || clip_left {
            0x3F00 // universal background color
        } else {
            // Pick bit 15 - x from each shifter
            let bit = 15 - self.regs.x as u32;
            let pat_lo = ((self.bg.shift_pat_lo >> bit) & 1) as u8;
            let pat_hi = ((self.bg.shift_pat_hi >> bit) & 1) as u8;
            let at_lo = (self.bg.shift_at_lo >> (7 - self.regs.x)) & 1;
            let at_hi = (self.bg.shift_at_hi >> (7 - self.regs.x)) & 1;
            let pat_bits = (pat_hi << 1) | pat_lo;
            if pat_bits == 0 {
                0x3F00
            } else {
                let at_bits = (at_hi << 1) | at_lo;
                0x3F00 + ((at_bits as u16) << 2) + pat_bits as u16
            }
        };
        let color = self.palette.read_masked(palette_addr, self.regs.greyscale());
        self.framebuffer[scanline * 256 + (dot - 1)] = color;
    }
}
```

### Task 32: OAMDMA at $4014 — 256-byte transfer with cycle alignment

**Background.** Writing to $4014 triggers a DMA transfer of 256 bytes from CPU memory `$XX00..$XXFF` (where XX is the value written) into PPU OAM. The transfer is implemented as 256 reads + 256 writes through `read_cpu_memory_no_tick` + `ppu.cpu_write(0x2004, val)`. Total cost: 513 cycles on even CPU alignment, 514 on odd (1 vs. 2 alignment cycles). The CPU is suspended for the duration; the bus advances cycles, the PPU runs, the APU runs.

The current bus has the $4014 case stubbed with `// OAMDMA: M2/M3 will implement the 256-byte transfer.`. Replace that.

- nesdev wiki — *PPU registers* §OAMDMA — https://www.nesdev.org/wiki/PPU_registers#OAMDMA
- Design spec §3.2

```rust
0x4014 => {
    // OAMDMA: 256 bytes from page (val << 8) → OAM.
    let page_base = (val as u16) << 8;
    // 1 cycle alignment for even CPU cycles, 2 for odd.
    self.tick();
    if self.cycles % 2 == 1 {
        self.tick();
    }
    for i in 0..256u16 {
        let addr = page_base + i;
        let byte = self.read_cpu_memory_no_tick(addr);
        self.tick();
        // Route through PPU's cpu_write to honor the OAMADDR auto-increment.
        self.ppu.cpu_write(&mut self.mapper, 0x2004, byte);
        self.tick();
    }
}
```

### Task 38: Sprite evaluation during dots 65-256 — find up to 8 in-range sprites

**Background.** During dots 65-256, the PPU walks primary OAM (64 entries × 4 bytes) and copies up to 8 in-range sprites into secondary OAM (32 bytes = 8 × 4). "In range" means: the sprite's Y coordinate satisfies `Y <= scanline < Y + sprite_height` (8 or 16 px). The walk is cycle-stealing — each pair of dots reads one byte and writes one byte.

We implement the *result* of the walk (what's in secondary OAM after dot 256) rather than the cycle-by-cycle process. The cycle-by-cycle is testable in principle but the test ROMs check the result, not the intermediate state.

**Important:** sprite-0 is the entry at OAM[0..4]. If sprite-0 ends up in secondary OAM (i.e., it's in range and within the 8-sprite limit), set `sprites.sprite_0_in_range = true` — needed for sprite-0 hit (Task 42).

References:
- nesdev wiki — *PPU sprite evaluation* §"Cycles 65-256" — https://www.nesdev.org/wiki/PPU_sprite_evaluation

```rust
impl Ppu {
    /// Walk primary OAM, find up to 8 sprites whose Y satisfies
    /// Y <= next_scanline < Y + sprite_height; copy them to secondary OAM
    /// in primary-OAM order. Run at dot 256 (after secondary clear).
    fn evaluate_sprites_for_next_scanline(&mut self) {
        let next_scanline = self.state.scanline as i16; // sprite Y is for *next* line: scanline N evaluates for scanline N+1
        let sprite_height = if self.regs.sprite_size_8x16() { 16 } else { 8 };

        self.sprites.count = 0;
        self.sprites.sprite_0_in_range = false;
        for slot in 0..8 { self.oam.secondary[slot * 4..slot * 4 + 4].fill(0xFF); }

        for i in 0..64 {
            let base = i * 4;
            let y = self.oam.primary[base] as i16;
            // Sprite displayed on scanlines y+1 .. y+sprite_height (Y is "previous scanline").
            // i.e., a sprite with Y=N first appears on scanline N+1.
            let row = next_scanline + 1 - (y + 1);
            if (0..sprite_height).contains(&row) {
                if self.sprites.count < 8 {
                    let dst = (self.sprites.count as usize) * 4;
                    self.oam.secondary[dst..dst + 4]
                        .copy_from_slice(&self.oam.primary[base..base + 4]);
                    if i == 0 { self.sprites.sprite_0_in_range = true; }
                    self.sprites.count += 1;
                } else {
                    // Overflow handled in Task 39.
                }
            }
        }
    }
}
```

> **Note on the y-coordinate semantics.** OAM Y is "the scanline of the top-pixel-row minus 1." A sprite with Y=$10 first appears on scanline 17. The formula `row = next_scanline + 1 - (y + 1)` simplifies to `row = next_scanline - y`. Use whichever is clearer in the code; the test ROMs are the spec.

### Task 39: Sprite overflow flag with the documented hardware bug

**Background.** When a 9th sprite would be placed in secondary OAM, the PPU sets PPUSTATUS bit 5 (sprite overflow) — but the hardware has a famous bug: it scans diagonally through the OAM after finding 8 sprites, treating subsequent bytes as Y values, so the bug produces false positives and false negatives. Specifically: after finding 8 sprites, OAMADDR increments by 5 instead of 4 each "miss," shifting the field by one each time. Most games avoid the overflow flag because of this bug.

For M2 we implement the simplified-but-still-buggy version that the test ROMs check: increment a phantom OAMADDR by 5 after sprite #9 is "found" by the buggy scan, set the flag if a 9th sprite's misinterpreted Y-byte is in range. The exact algorithm is documented on the nesdev wiki.

References:
- nesdev wiki — *PPU sprite evaluation* §"Sprite overflow bug" — https://www.nesdev.org/wiki/PPU_sprite_evaluation

### Task 40: Sprite fetch during dots 257-320 — pattern + attr + x into shifters

**Background.** During dots 257-320 (8 dots × 8 sprites = 64 dots), the PPU fetches each in-secondary-OAM sprite's pattern data. For sprite N (0-indexed), at dots 257 + N×8 through 264 + N×8:
- Dots 1-2 of the 8: garbage NT fetch (we ignore).
- Dots 3-4: garbage NT fetch (we ignore).
- Dots 5-6: pattern lo fetch.
- Dots 7-8: pattern hi fetch.

Pattern address depends on PPUCTRL bit 3 (8×8 mode) or the tile-LSB-as-pattern-table bit (8×16 mode). The pattern row within the sprite depends on `(scanline - sprite_y)`, with vertical flip per attribute bit 7.

We implement the result: at dot 320, all 8 sprite shifter slots are filled. Cycle-by-cycle accuracy isn't required for the test ROMs.

References:
- nesdev wiki — *PPU sprite evaluation* §"Cycles 257-320" — https://www.nesdev.org/wiki/PPU_sprite_evaluation

```rust
impl Ppu {
    /// At dot 320 of each visible/pre-render scanline, fetch pattern data
    /// for the up-to-8 sprites in secondary OAM and load the per-sprite
    /// shifters. Run for the *next* scanline.
    fn fetch_sprite_patterns(&mut self, mapper: &mut MapperKind) {
        let next_scanline = self.state.scanline as i16;
        let sprite_height: i16 = if self.regs.sprite_size_8x16() { 16 } else { 8 };

        for slot in 0..8 {
            let base = slot * 4;
            let y = self.oam.secondary[base] as i16;
            let tile = self.oam.secondary[base + 1];
            let attr = self.oam.secondary[base + 2];
            let x = self.oam.secondary[base + 3];

            let mut row = next_scanline - y;
            if attr & 0x80 != 0 {
                // Vertical flip
                row = sprite_height - 1 - row;
            }
            row = row.clamp(0, sprite_height - 1);

            let pat_addr_lo = if self.regs.sprite_size_8x16() {
                // 8x16 mode: pattern table is bit 0 of tile; tile pair is (tile & 0xFE).
                let pt = (tile as u16 & 1) * 0x1000;
                let tile_index = (tile as u16 & 0xFE) + (row >> 3) as u16;
                pt + (tile_index << 4) + (row & 7) as u16
            } else {
                let pt = self.regs.sprite_pattern_table_base();
                pt + ((tile as u16) << 4) + row as u16
            };

            let mut pat_lo = mapper.ppu_read(pat_addr_lo);
            let mut pat_hi = mapper.ppu_read(pat_addr_lo + 8);

            if attr & 0x40 != 0 {
                // Horizontal flip — reverse bits.
                pat_lo = pat_lo.reverse_bits();
                pat_hi = pat_hi.reverse_bits();
            }

            // For sprites past the in-range count, blank the shifter so the
            // pixel emit ignores them.
            if (slot as u8) >= self.sprites.count {
                pat_lo = 0;
                pat_hi = 0;
            }

            self.sprites.shifters[slot] = sprite::SpriteSlot {
                pat_lo, pat_hi, attr, x,
            };
        }
    }
}
```

### Task 41: Per-pixel sprite/background priority resolution

**Background.** Each visible dot, after the background pixel is emitted, the PPU checks each of the 8 sprite shifters: if `dot - sprite.x` is in 0..=7 (i.e., the sprite is on this column), pick that bit-row position from the pattern shifters; if non-zero, this is a *sprite pixel*. Lower-numbered sprites in secondary OAM win over higher-numbered (front priority). Then resolve sprite-vs-background:

- Sprite pixel transparent (pat bits = 00): show background.
- Background pixel transparent (pat bits = 00): show sprite.
- Both opaque, sprite has front priority (attr bit 5 = 0): show sprite.
- Both opaque, sprite has back priority (attr bit 5 = 1): show background.

PPUMASK bits 1 (show bg leftmost 8) and 2 (show sprite leftmost 8) and bits 3/4 (overall enables) interact with the result.

Sprite-0 hit (Task 42) is detected during this resolution: if pixel 0 of sprite 0 is opaque AND background pixel is opaque AND we're past the leftmost-8 clip, set PPUSTATUS bit 6.

References:
- nesdev wiki — *PPU rendering* §"Pixel output (background and sprites)" — https://www.nesdev.org/wiki/PPU_rendering

```rust
impl Ppu {
    fn emit_pixel(&mut self) {
        let scanline = self.state.scanline as usize;
        let dot = self.state.dot as usize;
        if scanline >= 240 || !(1..=256).contains(&dot) {
            return;
        }

        let bg_addr = self.bg_pixel_palette_addr(dot);
        let bg_opaque = (bg_addr & 0x03) != 0;
        let (sprite_addr_opt, sprite_back_priority, _sprite_was_zero) =
            self.sprite_pixel_palette_addr(dot);

        // Sprite-0 hit detection lands in Task 42 — until then, emit_pixel
        // just resolves the priority without latching the hit.

        let final_addr = match (bg_opaque, sprite_addr_opt) {
            (false, None) => 0x3F00,
            (false, Some(a)) => a,
            (true, None) => bg_addr,
            (true, Some(a)) => if sprite_back_priority { bg_addr } else { a },
        };

        let color = self.palette.read_masked(final_addr, self.regs.greyscale());
        self.framebuffer[scanline * 256 + (dot - 1)] = color;
    }

    /// Compute the palette address for the current background pixel,
    /// honoring leftmost-8 clip and global show_bg.
    fn bg_pixel_palette_addr(&self, dot: usize) -> u16 {
        if !self.regs.show_bg() || (!self.regs.show_bg_left8() && dot <= 8) {
            return 0x3F00;
        }
        let bit = 15 - self.regs.x as u32;
        let pat_lo = ((self.bg.shift_pat_lo >> bit) & 1) as u8;
        let pat_hi = ((self.bg.shift_pat_hi >> bit) & 1) as u8;
        let at_lo = (self.bg.shift_at_lo >> (7 - self.regs.x)) & 1;
        let at_hi = (self.bg.shift_at_hi >> (7 - self.regs.x)) & 1;
        let pat = (pat_hi << 1) | pat_lo;
        if pat == 0 { return 0x3F00; }
        let at = (at_hi << 1) | at_lo;
        0x3F00 + ((at as u16) << 2) + pat as u16
    }

    /// Compute the palette address for the highest-priority opaque sprite
    /// pixel at this dot. Returns (palette_addr, back_priority, was_sprite_0).
    fn sprite_pixel_palette_addr(&self, dot: usize) -> (Option<u16>, bool, bool) {
        if !self.regs.show_sprites() || (!self.regs.show_sprites_left8() && dot <= 8) {
            return (None, false, false);
        }
        for slot in 0..(self.sprites.count as usize) {
            let s = &self.sprites.shifters[slot];
            let sx = s.x as usize;
            if !(sx..sx + 8).contains(&dot.wrapping_sub(1)) {
                continue;
            }
            let bit = 7 - (dot - 1 - sx) as u32;
            let pat_lo = (s.pat_lo >> bit) & 1;
            let pat_hi = (s.pat_hi >> bit) & 1;
            let pat = (pat_hi << 1) | pat_lo;
            if pat == 0 {
                continue;
            }
            let palette = s.attr & 0b11;
            let addr = 0x3F10 + ((palette as u16) << 2) + pat as u16;
            let back_priority = s.attr & 0x20 != 0;
            let was_sprite_0 = slot == 0 && self.sprites.sprite_0_in_range;
            return (Some(addr), back_priority, was_sprite_0);
        }
        (None, false, false)
    }
}
```

### Task 42: Sprite-0 hit detection → PPUSTATUS bit 6

**Background.** Sprite-0 hit fires when, at any pixel of any visible scanline:
- The sprite-0 pixel is opaque (pat bits != 00).
- The background pixel is opaque (pat bits != 00).
- The pixel x is not 255 (the rightmost pixel does *not* trigger the hit — documented bug).
- The pixel x is not in the leftmost-8 clip region (when either the bg or sprite leftmost-8 is clipped, hit doesn't fire there either).
- Both background and sprite rendering are enabled (PPUMASK bits 3 AND 4 set).

The hit, once detected, latches PPUSTATUS bit 6 until cleared at scanline 261 dot 1 (already handled in Task 17).

### Task 47: OAMADDR corruption during rendering

**Background.** Documented hardware bug: OAMADDR writes during rendering (specific dot ranges of visible scanlines) corrupt OAM in a specific pattern.     // by copying the first 8 bytes of the addressed OAM "row" somewhere.
```rust
    // Per nesdev wiki, the canonical version is: bytes [val..val+8] are
```

### Task 48: A12 line plumbed through `mapper.notify_a12`

**Background.** Per design spec §3.5 of the global spec, the PPU notifies the mapper of A12 line transitions every PPU dot. NROM's `notify_a12` is a no-op, but the call site has to be there so MMC3 (post-v1) can hook in without PPU changes. A12 is bit 12 of the *current* PPU bus address — i.e., it's the high bit of which pattern table is being addressed.

For background fetches: A12 reflects the current pattern table being fetched (PPUCTRL bit 4 → bit 12 of the address). For sprite fetches: A12 reflects the sprite pattern table base. Outside rendering, A12 reflects v's bit 12 (during $2007 accesses through PPU bus).

Implementation: at the start of each `step()`, sample the "current address line" — really, the address that *would be* on the PPU bus at this dot if a fetch were happening. For our purposes (M2 with NROM), we model it as: during dots 1-256 of visible scanlines, A12 = `bg_pattern_table_base() >> 12` during the pattern fetches; during dots 257-320, A12 = sprite pattern table base. Otherwise, A12 = `(v >> 12) & 1`.

Since NROM ignores it, the exact dot precision doesn't matter at M2 — only that the call happens with a level value that *could* be correct for MMC3. We pick a simple model: `notify_a12((current_pattern_addr >> 12) & 1 != 0)` after each pattern fetch.

References:
- nesdev wiki — *PPU memory map* §"Pattern tables" — https://www.nesdev.org/wiki/PPU_memory_map
- Design spec §3.7 (`notify_a12` default no-op)

**Files:**
- Modify: `crates/nies-core/src/ppu/mod.rs`

- [ ] **Step 1: Add A12 notify after each PPU bus read.**

In the background-fetch arms (5, 7) and the sprite-fetch logic, after each `mapper.ppu_read(addr)` or `self.ppu_bus_read(...)` call where `addr < 0x2000` (CHR space), call:

```rust
mapper.notify_a12((addr >> 12) & 1 != 0);
```

Concretely, modify `ppu_bus_read`:

```rust
fn ppu_bus_read(&mut self, mapper: &mut MapperKind, addr: u16) -> u8 {
    let mirroring = mapper.mirroring();
    if addr < 0x2000 {
        let level = (addr >> 12) & 1 != 0;
        mapper.notify_a12(level);
        mapper.ppu_read(addr)
    } else if addr < 0x3F00 {
        self.vram.read(addr, mirroring)
    } else {
        self.palette.read(addr)
    }
}
```

For sprite fetches in `fetch_sprite_patterns`, the `mapper.ppu_read(pat_addr_lo)` calls also need to notify:

```rust

mapper.notify_a12(((pat_addr_lo + 8) >> 12) & 1 != 0);
let mut pat_hi = mapper.ppu_read(pat_addr_lo + 8);
```
